# お悩み相談AIアプリケーション

## 概要
ユーザーが日常の悩みや困りごとを気軽に相談でき、AIが親身になって回答してくれるWebアプリケーション。Claude APIを活用し、共感的で寄り添った対話体験を提供する。

## ターゲットユーザー
日常生活・仕事・人間関係などの悩みを気軽に誰かに相談したいが、身近に相談相手がいない、または匿名で相談したいと考えている人。

## 機能一覧

### Feature 1: 相談入力フォーム
- **説明**: ユーザーが悩みのテキストを入力し、送信できるフォーム
- **ユーザーストーリー**: ユーザーとして、自分の悩みをテキストで入力して送信したい。なぜなら、AIに相談内容を伝えるための手段が必要だから。
- **受け入れ基準**:
  - [ ] テキスト入力欄が画面上に表示されている
  - [ ] 入力欄にテキストを入力できる
  - [ ] 送信ボタンが表示されている
  - [ ] 空欄のまま送信ボタンを押すと、送信されずにエラーメッセージが表示される
  - [ ] テキスト入力後に送信ボタンをクリックすると、入力内容が送信される
  - [ ] 送信後、入力欄がクリアされる
- **優先度**: P0

### Feature 2: AI回答表示
- **説明**: 送信された相談内容に対して、Claude APIを使ったAIの親身な回答を画面に表示する
- **ユーザーストーリー**: ユーザーとして、相談内容を送信した後にAIからの回答を読みたい。なぜなら、悩みに対するアドバイスや共感を得たいから。
- **受け入れ基準**:
  - [ ] 相談送信後、回答が生成されるまでローディング表示が出る
  - [ ] AIの回答がチャット形式で画面に表示される
  - [ ] 回答内容は相談内容に対して共感的・親身なトーンである
  - [ ] エラー時（API通信失敗など）にはエラーメッセージが画面に表示される
- **優先度**: P0

### Feature 3: 会話履歴表示
- **説明**: 同一セッション内で複数回やり取りした相談と回答の履歴を、時系列で画面に一覧表示する
- **ユーザーストーリー**: ユーザーとして、過去のやり取りを画面上で振り返りたい。なぜなら、これまでの相談の流れを踏まえて追加の相談をしたいから。
- **受け入れ基準**:
  - [ ] ユーザーの相談メッセージとAIの回答メッセージが時系列で並んで表示される
  - [ ] ユーザーの発言とAIの発言が視覚的に区別できる（位置・色・アイコンなど）
  - [ ] 新しいメッセージが追加されると、自動で最新メッセージまでスクロールされる
  - [ ] 複数回のやり取り後も、過去のメッセージがすべて閲覧できる
- **優先度**: P0

### Feature 4: 相談カテゴリ選択
- **説明**: 相談前にカテゴリ（仕事、人間関係、健康、日常生活など）を選択できる機能
- **ユーザーストーリー**: ユーザーとして、相談のカテゴリを選択したい。なぜなら、カテゴリに応じたより的確なアドバイスを受けたいから。
- **受け入れ基準**:
  - [ ] カテゴリ選択UIが相談入力フォームの近くに表示されている
  - [ ] 少なくとも4つ以上のカテゴリが選択肢として存在する
  - [ ] カテゴリを1つ選択できる
  - [ ] 選択中のカテゴリが視覚的にハイライトされる
  - [ ] カテゴリ未選択でも相談を送信できる（任意選択）
  - [ ] 選択したカテゴリがAIの回答内容に反映される
- **優先度**: P1

### Feature 5: 相談アプリ向けデザイン・レイアウト
- **説明**: 温かみがあり安心感を与える、相談アプリケーションにふさわしいビジュアルデザイン
- **ユーザーストーリー**: ユーザーとして、落ち着いた温かみのあるデザインの中で相談したい。なぜなら、安心して悩みを打ち明けられる雰囲気が大切だから。
- **受け入れ基準**:
  - [ ] 暖色系または落ち着いたトーンの配色が適用されている
  - [ ] ページタイトルまたはヘッダーにアプリ名が表示されている
  - [ ] モバイル画面幅（375px）でもレイアウトが崩れずに操作できる
  - [ ] デスクトップ画面幅（1280px）でも適切なレイアウトで表示される
  - [ ] フォントサイズが読みやすい大きさである（本文14px以上）
- **優先度**: P0

### Feature 6: 新しい相談を始める機能
- **説明**: 現在の会話履歴をリセットし、新しい相談をゼロから始められる機能
- **ユーザーストーリー**: ユーザーとして、別の悩みについて新しく相談を始めたい。なぜなら、前の相談内容と混ざらないようにしたいから。
- **受け入れ基準**:
  - [ ] 「新しい相談を始める」ボタンが画面上に表示されている
  - [ ] ボタンをクリックすると会話履歴がクリアされる
  - [ ] クリア後、初期状態の画面に戻る
  - [ ] 会話がない初期状態では、ボタンが非表示または非活性になっている
- **優先度**: P1

### Feature 7: ウェルカムメッセージ
- **説明**: 初回アクセス時や新しい相談開始時に、AIからの挨拶・案内メッセージを表示する
- **ユーザーストーリー**: ユーザーとして、アプリを開いたときに案内メッセージを見たい。なぜなら、どう使えばいいか分かり、安心して相談を始められるから。
- **受け入れ基準**:
  - [ ] ページ読み込み完了時にウェルカムメッセージが画面に表示されている
  - [ ] メッセージ内に、相談の始め方や使い方のヒントが含まれている
  - [ ] 「新しい相談を始める」実行後にもウェルカムメッセージが再表示される
- **優先度**: P2

### Feature 8: 入力中のリアルタイムフィードバック
- **説明**: テキスト入力中に文字数カウントを表示し、入力状態を視覚的にフィードバックする
- **ユーザーストーリー**: ユーザーとして、入力中に文字数を確認したい。なぜなら、適切な長さで相談内容をまとめたいから。
- **受け入れ基準**:
  - [ ] テキスト入力中に現在の文字数がリアルタイムで表示される
  - [ ] 上限文字数が設定されており、上限に近づくと警告色に変わる
  - [ ] 上限文字数を超えると送信ボタンが非活性になる
- **優先度**: P2

### Feature 9: サーバープロセスの安定起動・待機 [変更: Sprint 3後]
- **説明**: `npm start` コマンドでサーバーが起動し、プロセスが終了せずにリクエストを待ち受け続ける。Windows PowerShell環境を含む全環境で安定動作する。
- **ユーザーストーリー**: 開発者・利用者として、`npm start` を実行するだけでサーバーが立ち上がり、落ちずに動き続けてほしい。なぜなら、手動でプロセスを維持する操作は不便だから。
- **受け入れ基準**:
  - [ ] `npm start` を実行するとサーバープロセスが起動する
  - [ ] サーバー起動後、プロセスが自動終了せず最低30秒以上待機し続ける
  - [ ] 待機中に http://localhost:3000 へHTTPリクエストを送ると、正常なHTTPレスポンス（ステータス200）が返る
  - [ ] サーバー起動時に「サーバーが起動しました」旨のメッセージがコンソールに表示される
  - [ ] Windows PowerShell環境で `node server.js` を実行してもプロセスが即終了しない
  - [ ] Ctrl+C でプロセスを正常終了できる
- **優先度**: P0

### Feature 10: 環境変数の自動読み込み [変更: Sprint 3後]
- **説明**: プロジェクトルートの `.env` ファイルから `ANTHROPIC_API_KEY` を自動的に読み込み、追加の設定手順なしでAI機能が利用できる。
- **ユーザーストーリー**: 利用者として、`.env` ファイルにAPIキーを書いておくだけで相談機能が使えるようにしたい。なぜなら、環境変数の手動設定は手間がかかるから。
- **受け入れ基準**:
  - [ ] `.env` ファイルに `ANTHROPIC_API_KEY=有効なキー` を記述した状態で `npm start` するとAI回答機能が正常動作する
  - [ ] `.env` ファイルが存在しない場合、サーバーは起動するがAI回答リクエスト時に分かりやすいエラーメッセージが返る
  - [ ] `ANTHROPIC_API_KEY` が空または未設定の場合、相談送信時に「APIキーが設定されていません」旨のエラーメッセージが画面に表示される
  - [ ] `.env` ファイル自体がバージョン管理に含まれていない（`.gitignore` に記載されている）
- **優先度**: P0

### Feature 11: デバッグログのクリーンアップ [変更: Sprint 3後]
- **説明**: 開発中に追加されたデバッグ用の冗長なログ出力を除去し、本番利用に適したクリーンなコンソール出力にする。
- **ユーザーストーリー**: 利用者として、サーバー起動時やリクエスト処理時に不要なデバッグ情報が大量に表示されないでほしい。なぜなら、重要な情報が埋もれて見づらくなるから。
- **受け入れ基準**:
  - [ ] サーバー起動時のコンソール出力が、起動完了メッセージとポート番号のみに限定されている
  - [ ] 相談メッセージの送受信時に、デバッグ目的の内部データ（リクエストボディ、APIレスポンス生データ等）がコンソールに出力されない
  - [ ] エラー発生時には、エラー内容を示すログが出力される（エラーログは残す）
- **優先度**: P1

### Feature 12: AI回答のストリーミング表示（タイピングアニメーション） [追加: Sprint 4後]
- **説明**: AIの回答を生成完了後に一括表示するのではなく、生成と同時に一文字（または小さなチャンク）ずつリアルタイムで画面に追記表示する。ユーザーは「AIが今まさに考えながら話してくれている」という没入感のある体験を得られる。
- **ユーザーストーリー**: ユーザーとして、AIの回答がタイピングされるようにリアルタイムで表示されてほしい。なぜなら、長い回答を待つ間の「沈黙」が減り、AIが寄り添ってくれている実感が増すから。
- **受け入れ基準**:
  - [ ] 相談送信後、ローディング表示（「回答を考えています...」等）が最初に出る
  - [ ] AIからの最初のテキストチャンクが到着した時点で、ローディング表示がAI回答メッセージ枠に切り替わる
  - [ ] AIの回答テキストが、一括で全文表示されるのではなく、徐々に増えながら画面に表示される（時間差で複数回スナップショットを取ると、表示文字数が増えていることが確認できる）
  - [ ] ストリーミング中はユーザーの相談入力欄および送信ボタンが非活性（無効化）になっており、追加送信できない
  - [ ] ストリーミング完了後、入力欄と送信ボタンが再び活性化する
  - [ ] ストリーミング完了後、最終的に表示されている回答全文が、従来の一括表示と同等の完全な内容である（途中で切れていない）
  - [ ] ストリーミング中も、新しいテキストが追加されるたびに会話履歴の最下部までスクロールが追従する
  - [ ] API通信エラーが発生した場合、従来と同様のエラーメッセージが画面に表示される（ストリーミング開始前・ストリーミング中のどちらで発生しても適切にエラー表示される）
  - [ ] ストリーミング応答を返すサーバーエンドポイントのレスポンスヘッダーが `text/event-stream`（SSE）である
  - [ ] ブラウザの開発者ツールのネットワークタブで、当該リクエストのレスポンスが chunked（段階的に受信される）ことが確認できる
  - [ ] ストリーミング表示は既存の全テーマ（Default / Ocean / Forest / Night / Sakura 等）で正常に動作する
  - [ ] ストリーミングで表示された回答も、完了後は会話履歴の一部として保持され、次の相談時に文脈として扱われる
- **優先度**: P1

### Feature 14: 感情記録UI（インライン絵文字セレクタ） [追加: Sprint 5後]
- **説明**: AIの各回答の直下に、5段階の絵文字セレクタ（😢😟😐🙂😊）をインラインで表示し、ユーザーが「今の気持ち」を1タップで記録できる機能。毎回の回答ごとに表示され、スキップ可能（未回答のまま次の相談を送信してもよい）。
- **ユーザーストーリー**: ユーザーとして、AIの回答を読んだ直後に今の気持ちを絵文字で記録したい。なぜなら、相談のたびに自分の気持ちがどう変わっているかを意識できると、相談の効果を実感しやすいから。
- **受け入れ基準**:
  - [ ] AIの回答メッセージの下に、5つの絵文字（😢😟😐🙂😊）がボタンとして横並びで表示される
  - [ ] 絵文字ボタンは、対応するAI回答が表示完了（ストリーミング完了）した後に表示される
  - [ ] 絵文字ボタンをクリックすると、そのボタンが「選択済み」として視覚的にハイライト表示される
  - [ ] 同じAI回答に対して選択済み絵文字を別の絵文字に変更できる（クリックし直すと上書きされる）
  - [ ] 絵文字を未選択のまま次の相談を送信しても、正常に送信でき、エラーにならない（スキップ可能）
  - [ ] 過去のAI回答それぞれに、個別に独立した絵文字セレクタが表示される（ある回答に選択した絵文字が他の回答の表示状態に影響しない）
  - [ ] 絵文字ボタンにはマウスホバー時のフィードバック（色変化・拡大等）があり、クリック可能なUIであることが視覚的に分かる
  - [ ] ストリーミング中のAI回答に対しては、絵文字セレクタはまだ表示されない（ストリーミング完了後に出現する）
- **優先度**: P0

### Feature 15: 気分に応じたAI回答トーン調整 [追加: Sprint 5後]
- **説明**: ユーザーが直近に記録した気分を Claude API のシステムプロンプトに組み込み、気分が低い（😢😟）ほど共感・傾聴寄りに、気分が高い（🙂😊）ほど前向き・サポート寄りに回答トーンを自動調整する機能。
- **ユーザーストーリー**: ユーザーとして、自分が落ち込んでいるときはAIに共感してほしく、元気なときは背中を押してほしい。なぜなら、そのときの気持ちに合ったトーンで返ってくる方が安心して相談を続けられるから。
- **受け入れ基準**:
  - [ ] ユーザーが 😢 または 😟 を選んだ直後の次の相談送信において、AI回答が共感・傾聴寄りのトーン（「つらかったですね」「その気持ち、よく分かります」等の共感表現を含む）で返る
  - [ ] ユーザーが 🙂 または 😊 を選んだ直後の次の相談送信において、AI回答が前向き・サポート寄りのトーン（「その調子です」「一緒に次の一歩を考えましょう」等の前向き表現を含む）で返る
  - [ ] ユーザーが 😐 を選んだ直後の次の相談送信において、AI回答が中立的でバランスの取れたトーンで返る
  - [ ] 気分未選択（セッション内でまだ一度も絵文字を押していない）のときは、従来どおりのモード設定（バランス/共感/解決）に従ったトーンで回答される
  - [ ] 気分に応じたトーン調整は、既存のモード選択（バランス/共感/解決）・カテゴリ選択と併存し、両方が反映される
  - [ ] Evaluator が「気分 😢 を記録 → 次の相談送信 → 回答内に共感表現が含まれる」というシナリオをPlaywright操作で再現・検証できる
- **優先度**: P0

### Feature 16: 本日の変化サマリカード [追加: Sprint 5後]
- **説明**: 「新しい相談を始める」ボタンを押したとき、即座にセッションをリセットせず、まず「本日の変化」サマリカードを表示する機能。セッション開始時の最初の気分、中盤の気分推移、最終気分を可視化し、ユーザーが相談前後の心境変化をメタ認知できる。確認後に実際のセッションリセットが実行される。
- **ユーザーストーリー**: ユーザーとして、相談を終えるときに「相談前はこんな気持ちだったけど、今はこう変わった」という変化を振り返りたい。なぜなら、相談の効果を自分で実感できると「相談してよかった」と思えるから。
- **受け入れ基準**:
  - [ ] 「新しい相談を始める」ボタンをクリックすると、会話履歴が即座にクリアされる前に、サマリカードがモーダルまたはオーバーレイとして表示される
  - [ ] サマリカードには「セッション開始時の気分」「セッション中盤の気分（1つ以上記録があれば）」「最終気分」の3ポイントが、それぞれ対応する絵文字で表示される
  - [ ] 気分が上昇した場合（例: 😢 → 🙂）、変化の方向性が視覚的に分かる（矢印、色、テキストなどいずれかで表現）
  - [ ] セッション中に気分記録が0件だった場合、「記録がありません」等のメッセージを表示し、それでもリセットは続行できる
  - [ ] セッション中に気分記録が1件のみの場合、「最初の気分のみ」と「最終気分のみ」として同じ絵文字が表示される、または1ポイントのみの簡略表示になる（いずれかの仕様で良い）
  - [ ] サマリカードには「リセットして新しい相談を始める」ボタンと「閉じる（リセットしない）」ボタンの2つが表示される
  - [ ] 「リセットして新しい相談を始める」を押すと、会話履歴と気分記録が全てクリアされ、初期状態に戻る
  - [ ] 「閉じる」を押すと、サマリカードが消えて元の相談画面に戻り、会話履歴も気分記録も保持される
  - [ ] Evaluator が「相談を複数回行い → 絵文字を複数回記録 → 新しい相談ボタン → サマリカードに気分推移が表示される」シナリオをPlaywright操作で再現・検証できる
- **優先度**: P1

### Feature 17: 既存機能のストリーミング下での互換維持 [追加: Sprint 4後]
- **説明**: ストリーミング表示の導入によって、既存の主要機能（カテゴリ選択、モード選択、テーマ切替、会話履歴、新しい相談を始める、文字数カウンタ）が破壊されず、これまでと同じように利用できること。
- **ユーザーストーリー**: ユーザーとして、タイピングアニメーションが加わっても今まで使っていた機能が変わらず動いてほしい。なぜなら、機能が退行すると相談体験が損なわれるから。
- **受け入れ基準**:
  - [ ] カテゴリを選択した状態で相談を送信すると、ストリーミングされた回答内容にカテゴリが反映されている
  - [ ] モードを選択した状態で相談を送信すると、ストリーミングされた回答のトーン／スタイルに選択モードが反映されている
  - [ ] テーマを切り替えた後にストリーミング中の回答が表示されても、レイアウト崩れや文字の重なりが発生しない
  - [ ] 「新しい相談を始める」ボタンをクリックすると、ストリーミング完了済みの会話履歴が全てクリアされ、初期状態に戻る
  - [ ] 入力中の文字数カウンタが、ストリーミング表示中・完了後のどちらでも正しく動作する（上限超過時の送信ボタン非活性も維持）
  - [ ] ストリーミング完了後、続けて次の相談を送信すると、前回のやり取りを踏まえた回答がストリーミング表示される（会話履歴の文脈が維持される）
- **優先度**: P0

### Feature 18: 匿名ユーザー識別とオンボーディング [追加: Sprint 5後]
- **説明**: 初回アクセス時に「ユーザー名」の入力を求めるオンボーディング画面を表示し、入力後にシステムが UUID を自動払い出し、`ユーザー名 + UUID` のペアをデータベースと永続化ストレージに保存する。2回目以降のアクセスでは保存済みの識別情報を自動利用し、オンボーディング画面はスキップされる。本機能はあくまで匿名識別であり、パスワード・メールアドレス等の認証情報は一切扱わない。
- **ユーザーストーリー**: ユーザーとして、初回だけ名前を入力すれば次回以降は自動で自分として認識されたい。なぜなら、毎回ログインする手間をかけずに、自分の過去の相談や気分の記録を継続的に残していきたいから。
- **受け入れ基準**:
  - [ ] 初回アクセス時（ブラウザに識別情報がない状態）で、相談画面の前にオンボーディング画面が表示される
  - [ ] オンボーディング画面にはユーザー名入力欄と「はじめる」等の確定ボタンが表示される
  - [ ] ユーザー名を空欄のまま確定ボタンを押すと、送信されずエラーメッセージが表示される
  - [ ] 有効なユーザー名を入力して確定ボタンを押すと、システムが UUID を自動生成し、ユーザー名 + UUID のペアがデータベースに保存される
  - [ ] オンボーディング完了後、自動的に相談画面に遷移する
  - [ ] 2回目以降のアクセス（識別情報がブラウザに保存済みの状態）では、オンボーディング画面がスキップされ直接相談画面が表示される
  - [ ] 相談画面のヘッダー等、画面上のいずれかの場所に現在のユーザー名が表示されている
  - [ ] オンボーディング画面および相談画面の全フローにおいて、パスワード・メールアドレス・電話番号等の入力は一切要求されない（匿名識別のみ）
  - [ ] Evaluator が「初回訪問 → ユーザー名入力 → 相談画面到達 → ページリロード → オンボーディングがスキップされ同じユーザー名で相談画面に到達」というシナリオをPlaywright操作で再現・検証できる
- **優先度**: P0

### Feature 19: 相談・気分データのDB永続化 [追加: Sprint 5後]
- **説明**: 相談メッセージ（ユーザー発言・AI回答）、感情記録（絵文字・記録日時・対応AI回答）、モード選択履歴（バランス/共感/解決）、カテゴリ選択履歴（仕事/人間関係/健康 等）を、匿名ユーザー識別（Feature 18のユーザー名 + UUID）に紐づけてデータベースに永続化する機能。ブラウザを閉じても、別日に再訪しても、同じユーザーとして過去データを読み出せる。
- **ユーザーストーリー**: ユーザーとして、今日の相談内容や気分記録が明日以降も残っていてほしい。なぜなら、過去の自分がどんなことに悩んでいたか、そのとき気分がどう変わったかを後から振り返れるようにしたいから。
- **受け入れ基準**:
  - [ ] 相談を1回送信するたびに、ユーザー発言の本文、AI回答の本文、送信日時、選択されていたモード、選択されていたカテゴリがデータベースに保存される
  - [ ] 絵文字を1回記録するたびに、選択された絵文字、記録日時、対応するAI回答への紐付けがデータベースに保存される
  - [ ] 保存される全データが、Feature 18で識別された匿名ユーザー（ユーザー名 + UUID）に紐づく
  - [ ] ブラウザを閉じて再度アクセスした際、同じユーザーとして識別され、過去の相談データ・気分記録が読み出せる状態になっている（Feature 20で可視化される）
  - [ ] 別のブラウザ／別のユーザー名で作成された別のUUIDのデータは、現在のユーザーからは参照できない
  - [ ] DB保存処理が失敗した場合でも、画面上の相談体験は継続でき、ユーザーに分かりやすいエラー表示が出る
  - [ ] Evaluator が「ユーザーAで相談＋気分記録 → ブラウザ再起動 → 同じユーザーAとして識別 → Feature 20 の履歴画面で過去データが確認できる」というシナリオをPlaywright操作で再現・検証できる
- **優先度**: P0

### Feature 20: 過去の相談履歴閲覧画面 [追加: Sprint 5後]
- **説明**: ヘッダーから「過去の相談履歴」画面に遷移でき、日付ごとに相談セッションが一覧表示される。一覧から各セッションを開くと、当時の相談内容（ユーザー発言・AI回答）と、そのセッション中の感情変化（セッション開始時／中盤／終盤）が確認できる機能。
- **ユーザーストーリー**: ユーザーとして、過去の自分の相談と気分の変化をいつでも振り返りたい。なぜなら、長期的に自分がどんな悩みを抱え、相談でどう気持ちが動いてきたかを俯瞰したいから。
- **受け入れ基準**:
  - [ ] 相談画面のヘッダー等に「過去の相談履歴」へ遷移するボタンまたはリンクが表示されている
  - [ ] ボタンをクリックすると、履歴一覧画面に遷移する
  - [ ] 履歴一覧画面では、過去の相談セッションが日付ごとに時系列（新しい順）で一覧表示される
  - [ ] 一覧の各セッション項目には、少なくとも「日付」と「最初の相談の冒頭テキスト（または要約）」が表示される
  - [ ] 一覧から任意のセッションをクリックすると、そのセッションの詳細画面に遷移する
  - [ ] セッション詳細画面では、当時のユーザー発言とAI回答が時系列で表示される
  - [ ] セッション詳細画面では、そのセッション中に記録された気分の変化（開始時・中盤・終盤の絵文字、または記録された全絵文字の推移）が可視化される
  - [ ] 履歴画面から相談画面に戻るナビゲーション（戻るボタン等）が用意されている
  - [ ] 過去の履歴閲覧中は、編集・削除ではなく「閲覧のみ」である（本スプリントでは編集・削除機能は持たない）
  - [ ] Evaluator が「相談を複数セッション実施 → 履歴画面へ遷移 → 日付一覧にセッションが並ぶ → セッションを開くと発言と気分推移が見える」というシナリオをPlaywright操作で再現・検証できる
- **優先度**: P1

### Feature 21: 中断した会話の再開プロンプト [追加: Sprint 6後]
- **説明**: アプリ再訪時に当日未クローズのセッションが存在すれば、「前回の続きから再開する / 新しく始める」の選択モーダルを表示する機能
- **ユーザーストーリー**: ユーザーとして、ブラウザを閉じた後に戻ってきたら前回の会話の続きをするか新しく始めるかを選びたい。なぜなら、途中だった相談を最初からやり直さずに済ませたいから
- **受け入れ基準**:
  - [ ] 当日の未 close セッションが DB に存在する状態で再訪すると、相談画面到達前に選択モーダルが表示される
  - [ ] モーダルには「前回の続きから再開する」「新しく始める」の2ボタンが表示される
  - [ ] 「前回の続きから再開する」を押すと、そのセッションの過去メッセージ（ユーザー発言・AI回答）が時系列でチャットエリアに再現表示される
  - [ ] 「前回の続きから再開する」を押すと、そのセッションの感情記録も復元され、各 AI 回答の直下に選択済み絵文字がハイライト表示される
  - [ ] 「前回の続きから再開する」後、新しい相談を送信すると同じ sessionId でメッセージが追記される（DB 上同一セッションに紐づく）
  - [ ] 「新しく始める」を押すと、前セッションが close され、新規セッションで初期状態の画面が表示される
  - [ ] 当日の未 close セッションが存在しない場合（前日以前のみ、または全て closed）、モーダルは表示されず通常通り相談画面に遷移する
  - [ ] 初回訪問（オンボーディング未完了）の場合は、本モーダルは表示されない（オンボーディング優先）
  - [ ] Evaluator が「相談送信 → ブラウザリロード → 再開モーダル表示 → 続きから再開 → 過去発言と感情記録が復元される」シナリオを Playwright で再現・検証できる
- **優先度**: P1

## スプリント計画

### Sprint 1: コア相談体験の構築
- Feature 1: 相談入力フォーム
- Feature 2: AI回答表示
- Feature 5: 相談アプリ向けデザイン・レイアウト
- **ゴール**: ユーザーが1回の相談を入力し、AIから親身な回答を受け取れる。相談アプリにふさわしい温かみのあるデザインが適用されている。

### Sprint 2: 継続的な対話体験
- Feature 3: 会話履歴表示
- Feature 6: 新しい相談を始める機能
- **ゴール**: 同一セッション内で複数回のやり取りができ、過去のメッセージを振り返れる。また、任意のタイミングで新しい相談を始められる。
- **依存**: Sprint 1の完了が前提（入力フォーム・AI回答・デザインが動作していること）

### Sprint 3: 体験の向上
- Feature 4: 相談カテゴリ選択
- Feature 7: ウェルカムメッセージ
- Feature 8: 入力中のリアルタイムフィードバック
- **ゴール**: カテゴリ選択でより的確な回答を得られ、ウェルカムメッセージと文字数表示によって使いやすさが向上している。
- **依存**: Sprint 2の完了が前提（会話履歴・新規相談機能が動作していること）

### Sprint 4: サーバー安定稼働とクリーンアップ [変更: Sprint 3後]
- Feature 9: サーバープロセスの安定起動・待機
- Feature 10: 環境変数の自動読み込み
- Feature 11: デバッグログのクリーンアップ
- **ゴール**: `npm start` 一発でサーバーが起動し、プロセスが終了せず待機し続ける。`.env`からAPIキーが自動読み込みされ、ブラウザからhttp://localhost:3000にアクセスして相談ができる。デバッグ用ログが除去されクリーンな状態である。
- **依存**: Sprint 3の完了が前提（全機能が実装済みであること）。本スプリントは既存機能の変更ではなく、サーバー起動の安定性確保とコード品質改善が目的。

### Sprint 5: ストリーミング表示による没入感向上 [追加: Sprint 4後]
- Feature 12: AI回答のストリーミング表示（タイピングアニメーション）
- Feature 17: 既存機能のストリーミング下での互換維持
- **ゴール**: AIの回答が一文字ずつリアルタイムで表示され、「AIが今まさに考えて話してくれている」没入感のある相談体験を実現する。同時に、既存の全機能（カテゴリ/モード/テーマ/会話履歴/新しい相談/文字数カウンタ）がストリーミング導入後も完全に動作する。
- **依存**: Sprint 4の完了が前提（サーバー安定稼働・API経由での相談送受信が動作していること）。サーバー側の相談応答エンドポイントはSSE（`text/event-stream`）で配信する構成に変更または新規追加する（具体的なエンドポイント設計はGenerator判断）。クライアント側は段階的にテキストを受信して画面に追記表示する。既存 `/api/consult` 相当の互換はGenerator判断で維持または置き換え可。

### Sprint 6: 感情トラッカー（セッション内体験） [追加: Sprint 5後]
- Feature 14: 感情記録UI（インライン絵文字セレクタ）
- Feature 15: 気分に応じたAI回答トーン調整
- Feature 16: 本日の変化サマリカード
- **ゴール**: ユーザーがAI回答ごとに「今の気持ち」を5段階絵文字で記録でき、記録した気分がAIの次回回答のトーンに反映される。さらに「新しい相談を始める」操作時にセッション内の気分推移をサマリカードで振り返れる。本スプリントは、セッション内で完結する感情トラッカー体験を成立させることを目的とし、DB永続化には踏み込まない（気分記録のDB保存は Sprint 7 の Feature 19 で扱う）。
- **依存**: Sprint 5の完了が前提（ストリーミング表示が動作しており、AI回答の表示完了タイミングを検出できること）。絵文字セレクタは AI回答のストリーミング完了後に出現する仕様のため、Sprint 5 の完了検出ロジックに依存する。

### Sprint 7: 匿名ユーザー識別とデータ永続化・会話継続 [変更: Sprint 6後]
- Feature 18: 匿名ユーザー識別とオンボーディング
- Feature 19: 相談・気分データのDB永続化
- Feature 20: 過去の相談履歴閲覧画面
- Feature 21: 中断した会話の再開プロンプト
- **ゴール**: 初回アクセス時のオンボーディングでユーザー名を取得し、システムが払い出したUUIDと組み合わせて匿名ユーザー識別を成立させる。相談メッセージ・感情記録・モード選択・カテゴリ選択をデータベースに永続化し、過去の相談履歴を日付ごとに閲覧できる画面を提供する。加えて、ブラウザを閉じた後の再訪時に当日の未完了セッションを明示的に再開できる導線を提供する。
- **依存**: Sprint 6の完了が前提（感情記録UIと気分データの扱いが成立していること）。Sprint 6までは気分記録がセッション内のみの保持だったが、本スプリントで Feature 19 によりDB永続化を導入し、Feature 20 の履歴画面で過去データを再生表示する。なお、本スプリントで導入するのは「匿名ユーザー識別」であり、パスワード等を伴う「認証・アカウント管理」は引き続きスコープ外である。

## 非機能要件
- AI回答の応答時間はローディング表示でカバーし、ユーザーを待たせている感覚を軽減する
- 主要ブラウザ（Chrome, Firefox, Safari, Edge 最新版）で動作する
- アクセシビリティ: キーボード操作で相談の送信が可能である（Enterキーでの送信対応）
- AIの回答トーンは共感的・親身であり、批判的・冷淡な表現を避ける
- [変更: Sprint 3後] Windows PowerShell環境（Windows 11）でサーバーが安定動作する
- [変更: Sprint 3後] `npm start` のみで全ての依存関係の読み込みとサーバー起動が完了する（追加の手動設定不要）
- [追加: Sprint 4後] ストリーミング表示は、主要ブラウザの標準機能（Fetch API + ReadableStream、または EventSource）で受信できる形式で配信する
- [追加: Sprint 4後] ストリーミングの初回チャンク到着までの体感待ち時間を、従来のローディング待ち時間と比較して短縮する（最初の文字が出るまでの時間を短くする）
- [追加: Sprint 5後] 感情記録UI（5段階絵文字）は、マウス操作・タッチ操作・キーボード操作のいずれでも選択できる（アクセシビリティ観点）
- [追加: Sprint 5後] 匿名ユーザー識別はパスワード・メールアドレス等の認証情報を一切扱わず、ユーザー名 + システム払い出しUUID の組でのみ識別する
- [追加: Sprint 5後] 相談メッセージ本文・気分記録・モード／カテゴリ履歴は全てデータベースに永続化され、ブラウザ再起動・別日アクセスでも同一ユーザーとしてデータを復元できる
- [追加: Sprint 6後] 再訪時の会話継続は当日のみを対象とする（前日以前のセッションは履歴画面からの閲覧のみ）

## スコープ外
- ユーザー認証・アカウント管理（パスワード・メールアドレス・SMSコード等を用いた本人認証は実装しない。Sprint 7 で導入されるのは「匿名ユーザー識別（ユーザー名 + 自動払い出しUUID）」であり、これは認証ではない）
- 相談履歴のエクスポート機能（CSV/PDF出力など）※閲覧機能は Feature 20 でスコープ内
- 過去の相談・気分記録の編集・削除機能（履歴は閲覧のみ）
- 複数端末間でのデータ同期（同一ユーザー名を別端末で入力しても、別UUIDが払い出され別ユーザーとして扱われる）
- 管理者画面
- 多言語対応（日本語のみ）
- プッシュ通知
- 他ユーザーとの相談共有機能
- 前日以前のセッションからの会話再開（履歴画面での閲覧のみ提供し、再開・追記はしない）
