## 重大所見サマリー
**ストリーミング機能は UI 上でまったく動作しない**。ユーザーが相談を送信しても AI 回答は 90 秒以上経過しても一切表示されず、ローディング表示（「回答を考えています...」）のまま停止する。送信ボタン・入力欄はずっと disabled のまま解除されず、ユーザーは追加の操作が一切できない「完全ハング状態」に陥る。

サーバー側の `req.on("close")` ハンドラが、クライアント接続が生きているにも関わらずリクエストボディ受信完了直後に発火し、`clientAborted = true` が立つ。その結果、

1. `for await (const event of stream)` 内のガード `if (clientAborted) break` により **最初のイベントで即 break**、delta が 1 件もクライアントに転送されない。
2. `if (!clientAborted)` 分岐が全てスキップされるため **`res.end()` が呼ばれず**、SSE レスポンスが開きっぱなしで終了しない。
3. クライアント側 `reader.read()` はチャンクが来ずブロックし続け、ユーザーは無限待機状態。

サーバー側ログで `req.on("close")` が delta 受信前に fire していることを実測で確認済み（診断ログ挿入テストの結果）。

## 発見したバグ

### バグ 1（Critical）: `req.on("close")` 誤発火によりストリーミング delta が一切送信されず、レスポンスが閉じられない
**場所**: `server.js:134-137`（`/api/consult/stream` エンドポイント内）
```js
req.on("close", () => {
  clientAborted = true;
});
```

**現象**:
- クライアントが接続を閉じていないにも関わらず、サーバーがレスポンス（SSE ヘッダ）を送信した直後に `req.on("close")` が発火し `clientAborted = true` になる。
- その直後、`for await (const event of stream)` の最初のイベント（`message_start`）の評価で `if (clientAborted) break` により即離脱。
- 続く `if (!clientAborted)` ブロックもスキップされるため `sendEvent("done", ...)` と `res.end()` が呼ばれず、SSE ストリームが開きっぱなしになる。
- クライアントの `reader.read()` はチャンクを待ち続け、ブラウザ UI は「回答を考えています...」のまま、入力欄・送信ボタンは永続 disabled のまま。

**再現手順**:
1. `node server.js` でサーバーを起動。
2. ブラウザで http://localhost:3000 を開く。
3. 入力欄に「最近疲れています」等の任意テキストを入れて「送信する」ボタンを押す。
4. ユーザーメッセージは右側吹き出しで表示される。
5. 「回答を考えています...」ローディングインジケータが表示される。
6. **90 秒以上経過しても AI 回答は表示されず、送信ボタン・入力欄は disabled のまま。ページを再読み込みしない限り復帰しない。**

**サーバー側診断結果（診断用ログを仕込んで再現した結果）**:
```
[stream] starting handler, clientAborted= false
[stream] stream obtained, entering for-await
[stream] req close event fired, clientAborted set true   ← ここで誤発火
[stream] got event type= message_start clientAborted= true
[stream] loop ended, deltaCount= 0 clientAborted= true
```
Anthropic SDK 単体（Express を介さない直接呼び出し）ではストリームが正常に 7 イベント返すことを確認済み。よって Anthropic API 側ではなく、Express の `req.on("close")` 発火タイミングが根本原因。

**期待動作**: delta イベントが逐次 SSE で送信され、クライアントに文字列が累積表示される。完了時に `event: done` と `res.end()` で正常終了。

**重大度**: **Critical**（P0 機能が完全に動作不能）

**修正指針**:
- `req.on("close")` の代わりに `res.on("close")` を使う。もしくは `res.writableEnded` / `res.destroyed` / `req.destroyed` で切断判定する。
- あるいは `abortSignal` を使い、Anthropic SDK の `stream()` に `signal` オプションを渡してキャンセル伝播させる。
- 参考: Express 4 では IncomingMessage の body 読み取り完了時に `close` イベントが発火する挙動が確認されており、サーバー処理完了前の `req.on("close")` は誤検知になりやすい。

### バグ 2（High）: 不整合な接続終了でクライアントが永遠にハングし、エラー表示にも到達しない
**場所**: `server.js:162-170`（`if (!clientAborted)` ブロック）

**現象**: バグ 1 と連動。`clientAborted = true` で `for await` が抜けた場合に `res.end()` が呼ばれない経路が存在する。クライアント側の `reader.read()` はタイムアウト無しで待機するため、UI が復帰する機会がない。

**期待動作**: サーバーが何らかの理由で正常完了に至らない場合でも、必ず `res.end()` を呼び、可能なら `event: error` を送信してクライアントがエラー表示・入力再活性化に進める。

**重大度**: **High**

### バグ 3（High）: ストリーミング開始前段階でユーザーが永続的に操作不能になる
**場所**: `public/app.js:297-303`（`finally` ブロック）

**現象**: サーバーが `res.end()` を呼ばない（バグ 1 の副作用）ため、クライアントの `try` ブロックが `await reader.read()` で無限ブロックされる。その結果 `catch`/`finally` に到達せず、`isStreaming = false` や `input.disabled = false` の復帰処理が実行されない。
- サンプル結果: `sendDisabled: true, inputDisabled: true` が 90 秒間固定。
- 2 回目以降の送信は `element is not enabled` でタイムアウト。

**期待動作**: サーバー応答が異常な場合でもクライアント側に一定時間の SSE タイムアウト（例: 60 秒の全体タイムアウト / 30 秒の idle タイムアウト）を設け、タイムアウト時にエラー表示＋入力再活性化を行う。

**重大度**: **High**

## 改善提案
1. `req.on("close")` を `res.on("close")` もしくは `res.closed` で置き換える（最優先の Critical 修正）。
2. `for await` 内の `if (clientAborted) break` 後でも、`res.writableEnded` でなければ `res.end()` を必ず呼ぶ終了処理を追加する（try 全体の `finally` ブロック推奨）。
3. クライアント側 `fetch` にタイムアウト（例: `AbortController` で 60 秒）を設定し、タイムアウト時にエラーメッセージを表示し入力欄を再活性化する。
4. ストリーミング開始前に、`reader.read()` の最初の読み取りで一定時間（例: 15 秒）経過してもデータが来ない場合にタイムアウト扱いとする。
5. サーバー側でストリーム開始後すぐに keep-alive コメント行（`: ping\n\n`）を 1 件送ることで、中継プロキシやクライアントが接続を認識しやすくする（将来拡張）。
6. Generator は今回の事例を踏まえ、「サーバー上で curl による SSE 実動作チェック」と「ブラウザ UI での実送信チェック」を commit 前に必ず行うこと。自己評価の「curl で `Content-Type: text/event-stream` を確認済み」はヘッダのみの確認で、body が 1 件も流れないことに気付いていなかった。

## 判定: ❌ 不合格

### 不合格理由
- 平均スコア **2.22 / 10**（閾値 7.0 を大きく下回る）
- 個別最低スコア **0**（閾値 4 を下回る項目が多数）
- P0 機能スコア **1.17 / 10**（閾値 8 を大きく下回る）
- ストリーミング機能の導入により既存機能も事実上利用不能（回帰破壊あり）

### 合格に必要な修正事項（Generator への指示）
1. **【Critical】`req.on("close")` を廃止または適切に置換する**
   - 推奨: `req.on("close")` ではなく、クライアント切断検知は `res.on("close")` を使うか、`abortSignal`（`AbortController`）をセットアップして Anthropic SDK の `stream({ signal })` に渡す。
   - 検証: 修正後、ブラウザで送信すると 1〜3 秒以内に `message-streaming` DOM 要素が現れ、その中の文字数が時間経過とともに増加することを Playwright 等で確認すること。

2. **【Critical】必ず `res.end()` が呼ばれる保証を入れる**
   - `try/finally` 構造で、成功・失敗・クライアント切断のどのパスでも `res.writableEnded` でなければ `res.end()` を呼ぶ。
   - 検証: `curl -N` で SSE レスポンスを取得したとき、Anthropic API 応答完了時にコネクションが閉じることを確認。

3. **【High】クライアント側にタイムアウトを導入**
   - 例: `fetch` に AbortController（60 秒）を設定する。`reader.read()` の idle タイムアウトも検討。
   - ストリーミング開始前に一定時間経過してもチャンクが来ない場合は「AIからの回答を取得できませんでした」等のエラーを画面に表示し、`isStreaming = false` と `input.disabled = false` にリセットする。
   - 検証: サーバー側でレスポンスを意図的に `res.end()` せず放置するスタブで、クライアントが所定時間後にエラー表示しつつ入力欄が再活性化することを確認。

4. **【Must】上記修正後に以下のシナリオを手元で全て通す**
   - a. 初回送信でローディング → 1〜3 秒以内にタイピング開始 → 文字が増加 → 完了後にカーソル消失 → 入力欄再活性化。
   - b. 連続 2 回の送信で、2 回目の回答が 1 回目の文脈を踏まえていること。
   - c. カテゴリ（例: 仕事）とモード（例: 解決）を選んだ状態で送信し、ストリーミング完了後の回答がカテゴリ・モードのトーンを反映していること。
   - d. 全テーマ（default/ocean/forest/night/sakura）でストリーミング中もレイアウト崩れが無いこと。
   - e. 「新しい相談を始める」で履歴がクリアされ、続いて新しい相談がストリーミング表示できること。
   - f. ストリーミング中は入力欄・送信ボタン・「新しい相談を始める」の 3 つ全てが disabled であり、完了後に全て再活性化されること。
   - g. 文字数 1001 文字で送信ボタン disabled、900 文字で warning クラスが付くこと（ストリーミング前/後いずれも）。

5. **【Must】修正後 `specs/progress.md` に「Sprint 5 不合格対応」セクションを追加**し、バグの根本原因・修正内容・再検証結果を記載してから Evaluator に再評価を依頼すること。
