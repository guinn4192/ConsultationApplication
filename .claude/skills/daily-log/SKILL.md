# デイリーログ記録スキル

1日の作業内容を構造化してmarkdownで `diary/` に残す。

## いつ使うか

- ユーザーが「今日の作業を記録して」「デイリー残して」と言った時
- セッション終盤に一区切りついた時
- 大きな機能追加・バグ修正・設計変更を完了した時

## 出力先

- パス: `diary/YYYY-MM-DD_<slug>.md`
  - `slug` はその日のメインテーマを英小文字ケバブケース（例: `theme-feature`, `mode-add`, `improvement-ideas`）
- 同日複数回なら `_part2` 等の枝番か別slugで分ける。既存ファイルを上書きしない
- 日付は `currentDate` から取得

## 記録前の情報収集

以下を並列で取得してから書き始める。

```bash
git status
git diff --stat
git log --oneline -20
```

- コミット済みの変更 → `git log` から拾う
- 未コミットの変更 → `git status` + `git diff --stat` から拾う
- 新規ファイル → `git status` の `??` 行

## テンプレート

```markdown
# <今日のテーマ 1行>

**日付**: YYYY-MM-DD
**関連スプリント**: Sprint N / なし

## やったこと

- <箇条書きで作業項目>
- <仕様追加 / バグ修正 / ドキュメント整備 等>

## 決めたこと

- <設計判断・トレードオフ>
- <採用した理由・却下した代案>

## ハマったこと / 解決

- <発生した問題>
  - 原因: <根本原因>
  - 解決: <対処内容>
- ハマりなしなら「特になし」

## 変更ファイル

- `path/to/file` — <何を変えたか>

## 次やること

- <次回着手する項目・TODO>
- <保留課題>
```

## ルール

1. **事実ベース**。実装していない機能を「やったこと」に書かない
2. **過程も残す**。試して却下した案、ハマった原因は将来の自分の資産
3. **ファイルパスは相対パス**でプロジェクトルート基準
4. **Why重視**。「何を変えたか」より「なぜ変えたか」を優先
5. **コミットしない**。ユーザーが明示指示するまでgit commitは走らせない
6. **既存 `diary/` の形式を尊重**。過去エントリのスタイルに合わせる
